# تقرير فحص الموقع — التوافق مع الهاتف ومعايير B2B

> آخر تحديث: 2026-06-01 · الفرع: `claude/b2b-mobile-audit-0xOcU` · PR: #11

هذه الوثيقة تشرح **على أي أساس** نفحص الموقع، **ما هي مستويات (محاور) الفحص**،
وما الذي **أُنجز** وما الذي **ما زال مطلوبًا** — مرتّبًا حسب الأولوية حتى تتمكن من
الموافقة على كل بند.

---

## 1) الأساس المعتمد للفحص (المراجع المعيارية)

الفحص والمقارنة لا يعتمدان على رأي شخصي، بل على معايير ومراجع عالمية معتمدة:

| المرجع | الجهة | ماذا يغطّي |
|--------|-------|-----------|
| **WCAG 2.2 — Level AA** | W3C | سهولة الوصول: تباين الألوان، أحجام اللمس، ARIA، النص البديل، التنقّل بلوحة المفاتيح |
| **Mobile-First Indexing / Mobile-Friendly** | Google | التجاوب، viewport، حجم الخط، عدم وجود تمرير أفقي |
| **Core Web Vitals** | Google | الأداء: LCP < 2.5s، CLS < 0.1، INP < 200ms |
| **Lighthouse** | Google / web.dev | تقييم آلي: Performance / Accessibility / Best Practices / SEO / PWA |
| **Human Interface Guidelines / Material Design** | Apple / Google | يعد المعيار لأهداف اللمس: ‎≥ 44px (Apple) / 48dp (Material) |
| **Open Graph Protocol** | Open Graph / Meta | بطاقات المشاركة على واتساب/لينكدإن/فيسبوك |
| **Schema.org (JSON-LD)** | Schema.org | البيانات المنظّمة لمحركات البحث (Organization / Product) |

> ملاحظة: المقارنة الحالية تتم مقابل **أفضل ممارسات B2B الصناعية** المبنية على المراجع
> أعلاه. إذا كان لديك رابط موقع B2B مرجعي محدّد تريد المقارنة معه مباشرة، زوّدني به
> وسأضيف عمود "الموقع المرجعي مقابل موقعنا" لكل بند.

---

## 2) مستويات (محاور) الفحص

نفحص الموقع عبر **7 محاور**، كل محور له معيار قياس واضح:

| # | المحور | ماذا نقيس | المرجع |
|---|--------|-----------|--------|
| 1 | **التجاوب التقني** | Breakpoints، viewport، لا تمرير أفقي، شبكات مرنة | Mobile-Friendly |
| 2 | **سهولة الوصول (A11y)** | تباين، أحجام لمس، ARIA، alt، focus | WCAG 2.2 AA |
| 3 | **الأداء** | LCP / CLS / INP، تحسين الصور، التحميل الكسول | Core Web Vitals |
| 4 | **SEO والميتا** | عناوين، وصف، OG، JSON-LD، sitemap/robots | Google / Schema.org |
| 5 | **تجربة B2B والتحويل** | اتصال بنقرة، نماذج عملاء محتملين، CTA، إشارات ثقة | أفضل ممارسات B2B |
| 6 | **التوافق عبر الأجهزة** | iOS/Android، RTL، متصفحات مختلفة، أحجام شاشات | Lighthouse / يدوي |
| 7 | **جودة الكود والأمان** | بناء نظيف، TypeScript، أسرار البيئة، RLS في Supabase | معايير هندسية |

## 4) ما تبقى — قائمة المهام (مرتّبة حسب الأولوية)

### 🔴 أولوية عالية

- [ ] **ربط نموذج "اتصل بنا" بإرسال حقيقي** — حاليًا تجريبي فقط ولا يرسل
      (`Contact.tsx`). الخيارات: Supabase Edge Function أو خدمة بريد (Resend/EmailJS).
      *المحور: B2B / التحويل — هذا أهم بند تجاري متبقٍّ.*
- [ ] **صورة OG حقيقية (1200×630 PNG)** بدل `logo.svg` — كثير من المنصّات لا تعرض
      صور SVG في بطاقات المشاركة. *المحور: SEO.*

### 🟠 أولوية متوسطة

- [ ] **منع قفزات التخطيط (CLS)** — إضافة `width`/`height` صريحة للصور (صورة
      "من نحن"، خلفية الـ Hero، صور المنتجات). *المحور: الأداء / Core Web Vitals.*
- [ ] **بيانات منظّمة JSON-LD** — `Organization` في كل الصفحات و`Product` في صفحة
      المنتج. *المحور: SEO.*
- [ ] **`sitemap.xml` و`robots.txt`** للأرشفة في Google. *المحور: SEO.*
- [ ] **`apple-touch-icon` + `manifest.webmanifest`** لإضافة الموقع للشاشة الرئيسية
      على الجوال (PWA أساسي). *المحور: SEO / موبايل.*

### 🟡 أولوية منخفضة (تلميع)

- [ ] **التحقق من تباين الألوان** — اللون الأخضر `brand-500 (#42d229)` على الأبيض
      أقل من 4.5:1 للنص الصغير؛ يُستخدم `brand-600` للنص (جيد) لكن يجب تدقيق كل
      الحالات. *المحور: A11y / WCAG AA.*
- [ ] **هياكل تحميل (skeletons)** بدل نص "טוען…" لتجربة أنعم. *المحور: UX.*
- [ ] **مفرد الـ Hero** — يوجد Hero ثابت + سلايدر؛ مراجعة إن كان التكرار مقصودًا
      على الجوال (طول الصفحة). *المحور: UX / موبايل.*
- [ ] **مفرّق المواصفات في صفحة المنتج** — `flex justify-between` قد يضغط القيم
      الطويلة على الشاشات الضيقة. *المحور: موبايل.*

## 6) كيف نتابع

1. راجع البنود في القسم (4) وحدّد أيّها تعتمده.
2. سأنفّذ المعتمد على نفس الفرع، مع `npm run build` ناجح قبل كل دفعة.
3. عند الانتهاء يمكن تشغيل **Lighthouse** للتأكيد الرقمي على الأهداف
   (Performance/Accessibility/SEO ≥ 90).
